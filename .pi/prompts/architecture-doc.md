<chain_of_thought>
  Before writing the ADD, reason through each of the following in a <zenith_analysis> block:

  1. TOPOLOGY — Architectural style (monolith/microservices/serverless/hybrid). Evidence from module list.
     Critical paths: which modules are depended on by the most others? Flag DAG cycles.

  2. BOUNDED CONTEXTS — Group modules by domain responsibility. Identify integration seams
     (REST API, event bus, library import, gRPC, etc.).

  3. OPERATIONAL POSTURE — Score each dimension:
     Observability: DARK | PARTIAL | INSTRUMENTED | FULL
     Deployability: NO_CONTAINER | PARTIAL | CONTAINERISED | PRODUCTION_READY
     Operability:   FRAGILE | STABLE | RESILIENT

  4. THREAT MODEL (STRIDE-lite) — For each bounded context, identify the top 2 security risks
     implied by the gap data. Flag any CRITICAL gaps that are active threats.

  5. REMEDIATION WAVES — Order by: CRITICAL first, then HIGH, then MEDIUM.
     Group into Construction Waves. Estimate effort per wave: LOW | MEDIUM | HIGH | VERY_HIGH.
</chain_of_thought>
